4) Fix the trust-killer: inconsistent stop / entry text

You flagged it yourself: one-line plan uses one stop, STOP box uses another.

Make a single source of truth for displayed stop/entry/target:

Rule
	•	Entry shown: signal.entryPriceAtActivation ?? signal.entryTriggerPrice ?? null
	•	Stop shown: signal.stopPrice ?? null
	•	T1 shown: tp?.t1 ?? signal.magnetPrice
	•	If those are missing, fallback to string fields (tp.entryTrigger, tp.invalidation) but label it clearly as “rule text.”

Quick patch

Replace the one-line plan rendering with a “data-first” line when live/activated:

const entryNum = signal.entryPriceAtActivation ?? signal.entryTriggerPrice ?? null;
const stopNum = signal.stopPrice ?? null;
const t1Num = tp?.t1 ?? signal.magnetPrice;

const oneLine =
  entryNum != null && stopNum != null
    ? `${biasLabel} → entry ${entryNum.toFixed(2)} → stop ${stopNum.toFixed(2)} → T1 ${t1Num.toFixed(2)}`
    : (oneLinePlan(signal) || `${biasLabel} → target $${signal.magnetPrice.toFixed(2)}`);

This prevents the UI from contradicting itself.
